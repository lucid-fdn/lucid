Complete reference for all environment variables used by Lucid.

## LLM / AI

Lucid is Lucid-first by default, but self-hosted deployments are not locked in.

- text and structured generation prefer Lucid Gateway when configured
- project / agent generation uses the same shared text-generation stack
- shared speech/transcription prefers Lucid-compatible media providers first, then falls back according to provider policy
- image generation uses the shared AI generation control plane with TrustGate/Lucid first and direct OpenAI only when explicitly enabled

At least one text-generation provider must be configured. Pick one option:

| Variable | Default | Description |
|----------|---------|-------------|
| `TRUSTGATE_API_KEY` | — | **Lucid Gateway** — preferred default for Lucid text/structured generation. One key, 100+ models |
| `OPENAI_API_KEY` | — | OpenAI API key (GPT-4, o3, etc.) |
| `ANTHROPIC_API_KEY` | — | Anthropic API key (Claude models) |
| `GROQ_API_KEY` | — | Groq API key (fast Llama/Mixtral) |
| `FALLBACK_PROVIDER_URL` | — | OpenAI-compatible endpoint (Ollama, vLLM, LM Studio) |
| `FALLBACK_PROVIDER_KEY` | — | API key for fallback provider |
| `FALLBACK_PROVIDER_MODEL` | — | Default model for fallback provider |
| `LUCID_API_BASE_URL` | `https://api.lucid.foundation` | Lucid / TrustGate base URL |

Provider architecture notes:
- text / structured generation: shared gateway resolves `TRUSTGATE_API_KEY` / `LUCID_API_KEY` first, then provider fallbacks such as `OPENAI_API_KEY`
- project / agent generation reuses that same stack rather than a separate provider path
- Lucid SDK-style services (memory, receipts, internal epoch jobs) use the shared Lucid provider config, not route-local env selection
- worker-side media and runtime paths also follow shared provider config helpers rather than channel-local env selection
- self-hosted precedence is Lucid-first when `TRUSTGATE_*` is configured, then explicit Lucid config, then BYOK fallbacks where the shared capability layer supports them

| `AI_GENERATION_IMAGE_ENABLED` | enabled | Rollout kill switch for image generation |
| `AI_GENERATION_AGENT_AVATAR_ENABLED` | enabled | Rollout kill switch for agent avatar generation |
| `AGENT_AVATAR_IMAGE_PROGRESS_ENABLED` | `true` | Enables provider partial-image streaming and persisted avatar job previews when the upstream supports it |

- direct OpenAI fallback is disabled by default for production TrustGate-only deployments

## Supabase Migration Operations

Lucid uses Supabase service-role keys for application reads/writes, but service-role keys cannot run DDL such as `ALTER TABLE`. Schema migrations need one of the DDL-capable paths below.

| Variable / Auth | Required For | Description |
|-----------------|--------------|-------------|
| Authenticated Supabase CLI | Preferred linked production migration path | Run `npx supabase db query --linked --file supabase/migrations/<file>.sql` from a repo linked to the target project. |
| `DATABASE_URL` | Direct Postgres migration | Full Postgres connection string with DDL privileges. |
| `SUPABASE_DB_URL` | Direct Supabase Postgres migration | Alias for a full Supabase Postgres connection string with DDL privileges. |
| `RAILWAY_DATABASE_URL` | Railway Postgres migration | Railway-provided Postgres URL when the target database is Railway-hosted. |
| `SUPABASE_DB_PASSWORD` | Linked Supabase pooler migration | Password for `supabase/.temp/pooler-url`; used by `npm run db:migration:apply` to construct a temporary connection string. |
| `SUPABASE_ACCESS_TOKEN` / `SUPABASE_PAT` | Supabase Management API migration | Personal access token for Supabase Management API migration calls. |
| `SUPABASE_PROJECT_REF` | Supabase Management API migration | Optional explicit project ref. If omitted, scripts infer it from `SUPABASE_URL` or `supabase/.temp/project-ref`. |

Canonical migration command:

```bash
npm run db:migration:apply -- --file supabase/migrations/<file>.sql --check-column <table.column>
```

The command first checks whether the migration is already visible through PostgREST, then tries direct Postgres, linked pooler password, Management API, and authenticated linked Supabase CLI. It prints `migration_status=already_applied` when the schema is live.

Worker AI/provider notes:
- current Railway production split uses `Lucid` for web/control plane, `lucid-channels` for channel gateways, `lucid-automation` for scheduled/background work, and `lucid-browser` for the isolated Browser Operator gateway
- production should split Railway worker services into `channels`, `worker`, `automation`, and `browser` process shapes once shared channels or Agent Ops are active
- `WORKER_MODE=all` logs a production warning and should not own channel sockets, queue workers, maintenance, and Browser Operator at the same time
- `/metrics.loop_ownership` reports which loops the current process owns
- `/metrics.pulse.queue_backlog_*` is the active pressure signal; legacy `/metrics.pulse.queue_depth_*` remains Redis stream length/history for compatibility
- worker media ingress and voice-reply flows use shared worker provider policy, not per-channel key selection
- Hermes runtime bootstrap and OpenClaw compatibility bootstrap reuse shared worker provider/env helpers
- some worker env vars remain runtime/control-plane specific by design (`LUCID_RUNTIME_*`, `LUCID_CONTROL_PLANE_URL`, `LUCID_L2_*`) and are not part of the generic inference-provider layer
- canonical local worker runtime: `cd worker && npm run start:local`
- production container/runtime environments also run the compiled worker path, not `tsx watch`

Policy behavior:
- shared SaaS should use a remote Browser Operator gateway or isolated `WORKER_MODE=browser` service
- local Playwright is blocked from generic shared worker mode unless the worker is explicitly running as an isolated browser worker
- Browserless, Stagehand, and remote-CDP provider details belong on the isolated browser gateway; shared workers should only see `BROWSER_QA_CONTROL_URL`
- Lucid-only mode is the safe default: leave `BROWSER_OPERATOR_EXTERNAL_PROVIDERS_ENABLED=false`, `BROWSER_OPERATOR_BYO_PROVIDERS_ENABLED=false`, and `BROWSER_OPERATOR_PREMIUM_FALLBACK_ENABLED=false`
- authenticated account sessions and commerce checkout are provider/profile pinned and must not silently fallback across providers
- Lucid owns merchant account, purchase policy, audit, idempotency, receipt, and memory/provenance state; browser providers are execution/session adapters only
- provider authentication can use Nango refs (`auth_provider`, `auth_connection_id`, optional `org_connection_id`) on Browser Operator accounts and BYO runtimes; Nango handles auth/token refresh, not browser execution
- Browserbase, Steel, Browserless, and Custom Browser Runtime are seeded into Settings through `plugin_catalog` as Nango/API-key integrations. They appear in the normal integrations UI after migrations run, but Connect requires matching provider configs in the centralized Nango service.
- default credential strategy is OAuth/delegated access or persistent browser sessions/profiles; Lucid-managed raw credentials are last-resort, feature-flagged, consented, audited, and gateway-only
- screenshots and large artifacts are stored by the gateway and referenced by URL
- usage/quota enforcement is backed by the Browser Operator usage ledger and plan-tier limits
- run `npm run agent-ops:browser-provider-smoke -- --require-live --run-session --target https://www.lucid.foundation` after changing browser provider envs
- in Lucid-owned production mode, the same smoke can run without hosted provider credentials: `BROWSER_QA_CONTROL_URL=https://<browser-gateway-host> npm run agent-ops:browser-provider-smoke -- --run-session --target https://www.lucid.foundation`
- `/mission-control/browser` is protected; complete account-health, secure-takeover, and alert-resolution UI smoke with an authenticated production session before marking that UI path fully verified
